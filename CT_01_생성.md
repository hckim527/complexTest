# complexTest — Jira 결함 테스트 샘플 생성 규칙

> **목적 (필독)**: complexTest 의 목적은 **Jira 에 등록된 현상(결함)의 재현에 그치지 않고, 그 현상을 넘어 추가로 검증이 필요한 부분을 발굴하여 테스트 케이스화함으로써 테스트 커버리지를 높이는 것**이다.
> - 따라서 모든 샘플은 ① 등록 현상의 재현/회귀에 더해 ② **현상 주변의 추가 검증 케이스**(경계값, 옵션·속성 조합, 연관 동작, SideEffect, 대조군 등)를 **반드시 함께 발굴·기재**한다. 등록 현상만 그대로 옮기는 것은 목적 미달이다.
> **워크플로우: CT_01_생성 → CT_02_검증 → CT_03_보강**
> - 본 문서로 샘플을 생성한 뒤 `complexTest/CT_02_검증.md` 규칙으로 검증한다.
> - 결함 재현/Jira 연계 특화 규칙은 `complexTest/CT_04_결함샘플.md` 를 함께 참조한다.

## 1. 대상 선정 및 입력

- **대상**: `[QA]` 프리픽스를 **제외**한 Jira 결함 중 **복합조건** 건. **해결(Open)** 건 을 대상으로 한다.
- **Jira 첨부 샘플의 컴포넌트를 그대로 사용**한다. 첨부의 검증된 구성을 써야 렌더/재현이 정확하다.
  - 첨부 다운로드: Jira API 토큰 basic-auth (`reference_jira_attachment_api_token` 참조). 첨부 id 는 `getJiraIssue` 의 `fields:["attachment"]` 로 조회.

## 3. 샘플 기본 구조 (setColumnIoFormat 패턴)

구조 레퍼런스: `unitTest/src/main/webapp/sample/gridView/gridView_M_setColumnIoFormat_1.xml` (구조만 참고).

- **head 메타**: `meta_screenName`, `historyInfo`, `xf:model` 등 표준 셸.
- **scwin.tData**: `description` + `validation` 배열.
  - `description` = **Jira 결함 제목 원문** + 줄바꿈(`<br/>`) 뒤에 **Jira 이슈 링크**. (`body_description` 은 `escape="false"` 라 HTML 렌더됨)
    - 형식: `"{Jira 제목}<br/><a href='https://inswave01.atlassian.net/browse/{KEY}' target='_blank' style='color:blue;text-decoration:underline;'>{KEY}</a>"`
    - 링크 **표시 텍스트는 이슈 키({KEY}, 예 `WAEA-937`)만**. 하이퍼링크 스타일로 `style='color:blue;text-decoration:underline;'`, `target='_blank'`(새 탭).
    - 앵커는 작은따옴표 속성 사용(description 은 큰따옴표 JS 문자열이므로 충돌 회피).
  - `validation` = **등록 현상의 회귀 + 현상 외 추가 검증 케이스**(상단 목적 참조). 기대/실제 서술이 아니라 검증 항목 형태로, Jira 현상 1건만 옮기지 말고 **인접 경계값·옵션 조합·연관 동작·SideEffect·대조군 등 추가 케이스를 발굴해 함께 기재**하여 커버리지를 넓힌다.
- **onpageload → comp_init → createValidation** 흐름:
  ```javascript
  scwin.onpageload = function () {
      $c.gcm.createCommonDatalist();
      scwin.comp_init();
      wf_body_left.getObj("scwin").createValidation(scwin.tData);
  };
  scwin.comp_init = async function () { /* 첨부 컴포넌트를 dynamicCreate */ }
  ```
- **body 5영역** (common_E/_M/_P 콘텐츠 wframe 미사용, `/frame/page/` 셸 사용):
  - `body_left` → `wf_body_left` (테스트 설명/유효성)
  - `grp_condition` → con_ 컨트롤 (Target 재생성 조건)
  - `wf_body_sample` → 테스트 샘플 (target 을 `$c.gcm.getTestGroup()` 에 dynamicCreate)
  - `grp_parameter` → par_ 컨트롤 + 실행 버튼(`grp_etc`)
  - `wf_body_bottom` → 리턴값 출력
- 화면 표시 정보(tData/라벨/리턴)와 **코드 주석**에 엔진 js 파일명·라인·내부 메서드명을 넣지 말 것. 원인 설명은 기능적 표현만.

## 4. ID/명명 규칙

- con_ 컨트롤(재생성 조건/Property): id = `con_{이름}`
- par_ 컨트롤(메소드 파라미터): id = `par_{이름}`
- 버튼 label 에는 파라미터 정보를 포함하지 않는다.
  - O: `label="setFooterStyle"`  /  X: `label="setFooterStyle(cellInfo, styleName, styleValue)"`
  - 단, 메소드 실행 버튼이 2개 이상이고 각 파라미터가 다르면 라벨에 파라미터를 포함한다.

---

## 5. UI 구성 규칙

- **con_/par_ 컨트롤(selectbox·input 등)은 `width`/`height` 를 설정하지 않는다** — `style=""` 로 두어 부모 th/td 영역에 autofit. (테스트 대상 target 컴포넌트는 예외이며 적절한 크기를 지정한다.)
- 옵션 항목(selectbox 등)은 한 줄(tr)에 2개까지만, `th, td, th, td` 쌍으로 생성.
- rowIndex/colIndex 등 옵션 값은 실제 행/컬럼 개수만큼 설정.
- con_/par_ 의 '미설정'(속성 미적용/기본값) 옵션은 **label/value 모두 빈값('')** 으로 작성.
- true/false 옵션 selectbox 는 `data:gdl_boolean` 데이터리스트를 바인딩:
  ```xml
  <xf:select1 ... id="par_visible" nodeSet="data:gdl_boolean" value="value">
      <xf:choices>
          <xf:itemset nodeset="data:gdl_boolean">
              <xf:label ref="label"></xf:label>
              <xf:value ref="value"></xf:value>
          </xf:itemset>
      </xf:choices>
  </xf:select1>
  ```

## 6. 출력값 작성 규칙

- 검증에 **직접 필요한 값만** `setReturnValue` 로 출력하여 불필요한 검증을 최소화한다.
- 디버깅용 부가 정보(컴포넌트 ID, 옵션 전체 dump, 중간 상태 등)는 출력하지 않는다.
- 한 줄로 충분한 정보를 여러 줄로 분할하지 않는다.
- **검증 항목 수 ≈ `setReturnValue` 호출 수** 가 일치하도록 작성.

## 7. 스크립트 작성 규칙

- DOM 요소 조회는 `document.getElementById` 대신 `document.querySelector` 사용.
  - 예: `document.querySelector("td[id*='target1_cell_0_0']")`
- set 형태 메소드는 정상 설정 여부를 확인하는 로직을 포함한다(쌍 get 메소드 호출, 없으면 DOM 에서 확인).
- 툴팁 텍스트 확인은 `$c.gcm.getTooltipInfo(wf_body_bottom)` 사용(직접 `.w2balloonTip_label` 접근 금지 — 잔존 풍선 DOM 오독 방지).
- 오늘 날짜는 `WebSquare.date.getCurrentServerDate()` 사용(기본 `yyyyMMdd`). `new Date()` 직접 생성 금지.

## 7-1. createLabel — 테스트 참고 라벨 규칙

> createLabel 관련 규칙은 분산하지 말고 본 절에만 둔다.

### 생성 시점·순서
- `comp_init` 에서 target 생성 후 상태가 변경되면 `$c.gcm.createLabel` 로 관련 내용을 textbox 라벨로 표시.
- `comp_init` 의 createLabel 라벨은 **순서상 가장 먼저 생성**(기존 target 삭제 후, 조건 설정 전).

### 라벨 내용 구성
- **필수 검증 내용만 간결하게.** 포함: ⓐ 현재 con 적용값(또는 고정 전제) ⓑ 검증 대상 동작 ⓒ 확인 방법(버튼/관찰 포인트). 제외: 동일 의미 반복, 자명한 설명.
- **validation 과 중복 금지** — 라벨은 validation 에 없는 운영 정보(전제 설정/조작 절차/확인 방법) 위주.
- **내부 구현 세부 금지** — 엔진 내부 동작/소스 경로/내부 메서드·컨트롤러 이름을 [참고]로 넣지 말 것. 사용자가 보는 동작 수준의 차이만 허용하고 그 원인(내부 로직)은 적지 않는다.
- 검증에 영향을 주는 상태 변경(숨겨진 컬럼, 고정 전제 등)만 명시.
- **줄바꿈으로 보기 좋게**: 여러 항목이면 `\n` 으로 줄을 나눈다(한 줄에 ` / ` 욱여넣기 금지). createLabel 이 `\n`→`<br/>` 변환.
  - 권장 구성: `[전제] ...\n[절차] ...\n[확인] ...`

### 문자 제약
createLabel 은 cList 가 **`label="..."` 단독 형식**이면 `setLabel`+`escape` 로 텍스트 주입. 이를 전제로:
- **그대로 사용 가능(안전)**: `<` `>` `&`, 라벨 내부 큰따옴표 `"`, 화살표 `->` `=>` `<-`.
- **금지/주의** (JS 문자열 리터럴 레벨이라 회피 필수):
  - **작은따옴표 `'`** : wrapper(`'label="..."'`) 조기 종료 → `SyntaxError`. 강조는 괄호 등으로.
  - **백슬래시+따옴표(`\'` `\"`)** : 변환 시 `SyntaxError`. 한글 설명으로 대체. `setReturnValue("...")` 등 JS 문자열에도 동일.
  - **HTML 태그(`<br/>` 등)** : escape 로 리터럴 표시되므로 직접 쓰지 말 것 → 줄바꿈은 `\n`.
- **단서**: cList 에 label 외 다른 속성을 함께 넘기면 escape 우회가 적용되지 않아 `<` `>` `"` 제약이 다시 살아난다.

## 8. 컴포넌트 배치 규칙

- `grp_comp` 하위에 컴포넌트를 배치해야 하면, `wf_body_sample` wframe 을 삭제하고 `body_sample.xml` 의 `grp_sample > grp_comp` 구조를 직접 배치한다.
  ```xml
  <xf:group style="" id="grp_sample">
      <w2:textbox style="" id="" label="■ 테스트 샘플" class="tc_tbx_label_1" escape="false"></w2:textbox>
      <xf:group style="" id="grp_comp" class="tc_group_basic">
          <!-- 정적 컴포넌트 배치 -->
      </xf:group>
  </xf:group>
  ```
